# 要件定義書 — うちの子カルテ

## サイト概要（サイト名・目的・想定ユーザー・課題・提供価値・最終ゴール・収益モデル・運用者・初期リリース範囲）

- サイト名：うちの子カルテ
- 目的：猫・犬の毎日の小さな変化を1分で残し、動物病院で「いつから？どんな感じ？」に落ち着いて答えられる状態を作る。
- 想定ユーザー：猫・犬を飼っているが、体調変化を獣医にうまく伝えられず、受診時に罪悪感や焦りを感じた経験がある飼い主。
- 表面的な課題：食事量、排便、気分、気になる症状を毎日記録したい。
- 本質的な課題：獣医に聞かれたとき、記憶だけでは変化の始まりや頻度を説明できず、「ちゃんと見ていたはずなのに答えられない」と感じてしまう。
- 提供価値：記録を増やすことではなく、受診前に「いつもと違う」が整理されたレポートを持参できる安心感を提供する。
- 最終ゴール：飼い主が獣医に対して、過去2週間の変化を日付・傾向・異常ポイントつきで堂々と説明できるようにする。
- 収益モデル：コンテスト応募作のため初期リリースでは収益化しない。将来は任意寄付、印刷テンプレート拡張、プレミアムテーマなどを検討可能。
- 運用者：darari本人。
- 初期リリース範囲：1ページの静的Webサイト。HTML、CSS、Vanilla JSで構成し、API、サーバー、ログイン、外部データ送信は行わない。対象ペットは猫・犬。データはブラウザのlocalStorageに保存する。

## サイト種別チェック（情報サイト/運用型/Webアプリ の判定。13項目チェックリストを使う：情報を見せるだけか、問い合わせ、販売、会員、決済、管理画面、外部連携、定期処理、個人情報保存、複数人運用、メール通知、AI機能、UGC）

判定：1ページ完結のローカル保存型Webアプリ。情報サイトではなく、ユーザーが日々データを入力し、受診用レポートを生成する道具として扱う。ただし、サーバー運用を伴う運用型サービスではない。

## ユーザー導線（初回訪問→日次入力→獣医前のPDF出力の3フロー）

### 初回訪問フロー

1. トップを開き、「うちの子カルテ」が記録アプリではなく、獣医に伝えるための準備サイトであることを理解する。
2. ペット名、種類、年齢または誕生日、通常の食事量、通常の排便回数、メモ欄に持病や薬を任意入力する。
3. 保存後、今日の記録セクションへ移動し、その日の状態を入力できる。
4. データが端末内だけに保存されることを設定セクションで確認できる。

### 日次入力フロー

1. サイトを開くと、今日の日付と選択中のペットが表示される。
2. 食事量を「完食」「少なめ」「半分以下」「食べない」からタップする。
3. 排便を「通常」「少ない」「多い」「下痢」「出ていない」からタップする。
4. 気分を「元気」「普通」「眠い」「不機嫌」「ぐったり」からタップする。
5. 気になることを「吐いた」「咳」「かゆい」「歩き方」「水をよく飲む」「その他」などの絵文字ボタンで選ぶ。
6. 必要な場合だけ短いメモを追加し、保存する。
7. 保存後、今日の記録カードに「受診で伝えやすい形で保存済み」と表示する。

### 獣医前のPDF出力フロー

1. レポート出力セクションで対象ペットと期間を確認する。初期値は過去14日間。
2. システムが通常値との差分を集計し、「いつもと違う」日を赤くハイライトする。
3. 過去2週間の食事、排便、気分、気になることを日付順にまとめる。
4. 「印刷ビューを開く」ボタンでA4想定の受診用レポートを表示する。
5. ブラウザの印刷機能でPDF保存または紙印刷する。

## 公開ページ（トップ1ページ。セクション構成：今日の記録 / 過去カレンダー / レポート出力 / 設定）

公開ページは`index.html`の1ページのみとする。ページ内アンカーで主要セクションに移動できる。

### 今日の記録

- 選択中ペット名と今日の日付を表示する。
- 食事量、排便、気分、気になることをタップ中心で入力する。
- メモ欄は任意で、プレースホルダーは「獣医さんに聞かれそうなことを一言だけ」など、長文を求めない表現にする。
- 保存後は同じ日の再編集が可能。

### 過去カレンダー

- 直近14日を標準表示し、必要に応じて30日まで確認できる。
- 記録済みの日は丸印、いつもと違う日は赤いマークで示す。
- 日付を押すと、その日の食事、排便、気分、気になること、メモが展開される。

### レポート出力

- 過去2週間の変化サマリーを表示する。
- 「いつもと違う」検出結果を赤いラベルで強調する。
- 印刷ビューでは、獣医が読みやすいように日付、項目、変化内容、メモを簡潔な表にする。
- PDF生成は専用ライブラリを必須にせず、ブラウザの`window.print()`を使ったPDF保存をv1の標準とする。

### 設定

- ペットプロフィールを登録、編集する。
- 通常の食事量、通常の排便回数、普段の気分を設定する。
- v2以降で複数ペット切替を強化するが、データ設計は初期から複数ペット対応にする。
- データ削除ボタンを設け、削除前に確認ダイアログを表示する。

## データ設計（ローカルストレージのkey-valueスキーマ。例：pets/{id}/profile, pets/{id}/records/{date}, settings）

localStorageは同一オリジン単位で保持される文字列key-valueストアとして使う。値はJSON.stringifyした文字列で保存し、読み込み時にJSON.parseする。

### key一覧

| key | value | 用途 |
| --- | --- | --- |
| `uchinoko:version` | `"1"` | データスキーマのバージョン。 |
| `uchinoko:pets` | `["pet_abc123", "pet_def456"]` | 登録済みペットIDの配列。 |
| `uchinoko:pets/{id}/profile` | `PetProfile` | ペットプロフィールと通常値。 |
| `uchinoko:pets/{id}/records/{date}` | `DailyRecord` | 1日分の記録。dateは`YYYY-MM-DD`。 |
| `uchinoko:settings` | `Settings` | 選択中ペット、表示期間、テーマなど。 |

### 値の候補

- `species`：`cat`、`dog`。将来拡張として`bird`、`small_animal`を追加できるよう文字列enumにする。
- `meal`：`normal`、`less`、`half`、`none`。
- `poop`：`normal`、`less`、`more`、`diarrhea`、`none`。
- `mood`：`active`、`normal`、`sleepy`、`grumpy`、`weak`。
- `concerns`：`vomit`、`cough`、`itchy`、`limp`、`drinks_more`、`pee_issue`、`other`。

## 業務ロジック（『いつもと違う』検出のルール、PDF生成のロジック、複数ペットの切替）

### 「いつもと違う」検出ルール

「診断」ではなく、飼い主が獣医に伝えるべき変化を見つける補助として扱う。

- 食事量：`half`または`none`は赤ハイライト。`less`が2日連続した場合も赤ハイライト。
- 排便：`diarrhea`、`none`は赤ハイライト。`less`または`more`が3日中2回以上あれば注意ハイライト。
- 気分：`weak`は赤ハイライト。`sleepy`または`grumpy`が2日連続した場合は注意ハイライト。
- 気になること：`vomit`、`cough`、`limp`、`pee_issue`は赤ハイライト。`drinks_more`が2日以上出た場合も赤ハイライト。
- メモ：メモ本文に`血`、`痛`、`震え`、`呼吸`、`ぐったり`が含まれる場合は確認ラベルを付ける。ただし医療判断として断定しない。
- 未入力日：異常として扱わない。レポート上では「記録なし」として表示する。

### レポート生成ロジック

1. 対象ペットの`profile`と、対象期間の`records`を取得する。
2. 日付を昇順に並べる。
3. 各日の記録に`detectFlags(record, previousRecords, profile.baseline)`を適用する。
4. サマリーとして、食事低下日数、排便異常日数、気分低下日数、気になることの頻出項目を集計する。
5. 獣医向けに、最初に変化が出た日、連続日数、直近の状態を短文で生成する。
6. 印刷ビュー用DOMに、プロフィール、2週間サマリー、日別表、メモ、注記を描画する。
7. `window.print()`で印刷ダイアログを開く。PDF保存はブラウザ標準機能に任せる。

### 複数ペットの切替

- `uchinoko:pets`に登録順でIDを保持する。
- 画面上部のセレクトまたはタブで`activePetId`を切り替える。
- 今日の記録、カレンダー、レポートは常に`activePetId`を基準に表示する。
- v1では1匹登録を主導線にし、v2で複数ペット追加UIを拡張する。

## 外部サービス連携（なし）

外部サービス連携は行わない。クラウド保存、アクセス解析、広告タグ、地図、SNSログイン、AI API、メール配信は初期リリース範囲外とする。個人情報外部送信ゼロを価値として明確に打ち出す。

## セキュリティ（ローカル完結なので情報漏洩なし、ただしブラウザのデータをエクスポート時の注意）

- データはlocalStorageに保存し、サーバーへ送信しない。
- 飼い主の氏名、住所、電話番号、メールアドレスは入力項目にしない。
- ペット名や健康メモは個人的な情報になり得るため、共用端末では利用者自身が注意する。
- ブラウザの履歴削除、サイトデータ削除、シークレットモードではデータが消える可能性があることを設定セクションに明記する。
- 将来のエクスポート機能では、JSONファイルやPDFを第三者に共有する前に内容を確認する注意文を表示する。
- XSS対策として、メモなどユーザー入力をHTMLとして挿入せず、`textContent`で描画する。

## インフラ（GitHub Pages）

- ホスティング：GitHub Pages。
- 構成：静的ファイルのみ。`index.html`、`style.css`、`script.js`、画像アセット。
- ビルド：不要。Vanilla JSで実装し、npm依存を持たない構成を優先する。
- 公開元：GitHub Pagesの設定で`main`ブランチのルート、またはリポジトリ構成に応じて`/docs`を選ぶ。応募用サイト本体はルート公開を想定する。

## デプロイ（mainブランチ→GitHub Pages自動。Actionsは不要）

1. `main`ブランチに静的ファイルを配置する。
2. GitHubのSettings > Pagesで、Build and deploymentを`Deploy from a branch`にする。
3. Branchを`main`、Folderを`/ (root)`に設定して保存する。
4. 以後、`main`へpushするとGitHub Pagesが自動更新される。
5. 専用ビルドが不要なため、GitHub Actionsのカスタムworkflowは作成しない。

## 運用（darari本人。記録ペース設計、データ移行）

- 運用者はdarari本人。
- コンテンツ更新は、文言、デザイン、説明注記、ロードマップの調整が中心。
- 利用者の記録データは運営者が取得しないため、問い合わせ対応やデータ復旧は行えない。
- 記録ペースは「毎日完璧に書く」ではなく「1分で異変の手がかりを残す」設計にする。
- データ移行はv3でJSONエクスポート/インポートを追加し、端末変更時に利用者自身が移せるようにする。

メール通知は実装しない。メールアドレスを取得しないことで、個人情報管理と外部送信の負担を避ける。リマインダーが必要な場合は、スマホ標準のカレンダーやリマインダーアプリを併用してもらう案内に留める。
